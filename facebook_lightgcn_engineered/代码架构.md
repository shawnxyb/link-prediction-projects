# LightGCN 工程代码架构说明

这个工程是把原来的单文件 `train_lightgcn.py` 拆成了多个模块，核心逻辑仍然是：**读取 Facebook 图数据 → 划分边 → 负采样 → 构建 LightGCN → 训练 → 验证 → 测试 → 保存结果**。原始单文件版包含这些逻辑。

---

## 1. 整体目录结构

```text
facebook_lightgcn_engineered/
├── train.py                    # 程序入口
├── requirements.txt            # 依赖库
├── README.md                   # 使用说明
└── src/
    ├── __init__.py             # 标记 src 为 Python 包
    ├── config.py               # 训练参数配置
    ├── data_utils.py           # 数据读取、划分、负采样
    ├── graph_utils.py          # 图结构工具
    ├── model.py                # LightGCN 模型
    ├── metrics.py              # 评价指标
    ├── trainer.py              # 训练主流程
    └── utils.py                # 通用工具函数
```

---

# 2. 程序运行主线

运行：

```bash
python train.py --data_path facebook_combined.txt
```

实际执行流程是：

```text
train.py
    ↓
parse_args()
    ↓
TrainConfig
    ↓
run_train(cfg)
    ↓
read_edges()
    ↓
make_split()
    ↓
build_norm_adj()
    ↓
LightGCN()
    ↓
训练循环
    ↓
evaluate_auc_ap()
    ↓
evaluate_ranking()
    ↓
保存模型、数据划分、实验结果
```

# 5. `data_utils.py`：数据处理模块

这是整个工程里最重要的模块之一。

---

## 5.1 类型定义

### `Edge = Tuple[int, int]`

表示一条边：

```python
(u, v)
```

---

## 5.2 数据划分类

### `EdgeSplit`

```python
@dataclass
class EdgeSplit:
```

作用：保存一次完整的数据划分结果。

包含：

```text
train_edges
val_edges
test_edges
val_neg_edges
test_neg_edges
num_nodes
```

也就是：

|字段|含义|
|---|---|
|`train_edges`|训练正边|
|`val_edges`|验证正边|
|`test_edges`|测试正边|
|`val_neg_edges`|验证负边|
|`test_neg_edges`|测试负边|
|`num_nodes`|节点数量|

---

## 5.3 边标准化

### `normalize_edge(u, v)`

作用：把无向边统一成：

```text
u < v
```

例如：

```python
normalize_edge(5, 2)
```

返回：

```python
(2, 5)
```

目的：

> 避免 `(u,v)` 和 `(v,u)` 被当成两条不同的边。

---

## 5.4 读取数据

### `read_edges(path)`

作用：读取 `facebook_combined.txt`。

处理流程：

```text
1. 逐行读取 u v
2. 去除自环
3. 无向边标准化
4. 去重
5. 将原始节点编号重新映射为 0 ~ num_nodes-1
6. 返回 edges, num_nodes, node2id
```

返回：

```python
edges, num_nodes, node2id
```

其中：

|返回值|含义|
|---|---|
|`edges`|边数组|
|`num_nodes`|节点数|
|`node2id`|原始节点 ID 到新 ID 的映射|

---

## 5.5 并查集

### `DSU`

作用：用于划分数据时保留训练图的连通骨架。

主要函数：

```python
find(x)
union(a, b)
```

用途：

> 在划分 train / val / test 时，先保留一个生成森林到训练集，减少训练图中孤立节点数量。

---

## 5.6 划分正边

### `split_edges_preserve_connectivity(...)`

作用：将真实边划分为：

```text
train_edges
val_edges
test_edges
```

核心思想：

```text
1. 打乱所有边
2. 用并查集保留一部分边作为生成森林
3. 这些边强制放入训练集
4. 剩余边再划分为验证集、测试集、训练集
```

为什么这么做？

> 如果直接随机划分，有些节点可能在训练图中变成孤立点，LightGCN 就无法从邻居中学习它们的表示。

---

## 5.7 构建边集合

### `build_edge_set(edges)`

作用：把边数组转成 Python `set`。

例如：

```python
{(0, 1), (1, 3), (2, 5)}
```

用途：

- 快速判断一条边是否存在；
    
- 负采样时避免采到真实边。
    

---

## 5.8 负采样

### `sample_negative_edges(...)`

作用：从完整图的非边中采样负样本。

正样本：

```text
(u, v) ∈ E
```

负样本：

```text
(u, v) ∉ E
```

关键点：

```python
full_edge_set
```

必须来自完整图，而不是训练图。

原因：

> 验证边和测试边虽然不在训练图里，但它们仍然是真实边，不能被误采成负样本。

---

## 5.9 一键生成划分

### `make_split(...)`

作用：封装完整数据划分流程。

内部会调用：

```text
build_edge_set()
split_edges_preserve_connectivity()
sample_negative_edges()
```

最终返回：

```python
EdgeSplit(...)
```

---

## 5.10 保存划分

### `save_split(split, path)`

作用：把数据划分保存成 `.npz` 文件。

输出文件：

```text
facebook_split_seed2024.npz
```

这个文件很重要，因为 MLP 组员也应该使用同一份划分，保证公平对比。

# 6. `graph_utils.py`：图结构工具模块

---

## 6.1 构建归一化邻接矩阵

### `build_norm_adj(num_nodes, train_edges, device)`

作用：构建 LightGCN 使用的归一化邻接矩阵。

LightGCN 传播公式：

# $$  
e_u^{(k+1)}

\sum_{v\in N(u)}  
\frac{1}{\sqrt{d_u d_v}}e_v^{(k)}  
$$

这个函数构造的就是：

# $$  
\hat A_{uv}

\frac{1}{\sqrt{d_u d_v}}  
$$

注意：

> 这里只能使用 `train_edges`，不能使用验证边和测试边。

否则会产生信息泄漏。

---

## 6.2 构建邻居表

### `build_neighbors(num_nodes, edges)`

作用：根据边构建邻居集合。

例如：

```python
neighbors[0] = {1, 3, 5}
```

用途：

- Top-K 推荐时排除训练集中已经是好友的人；
    
- 计算测试集中每个用户的真实好友集合。
    

---

# 7. `model.py`：LightGCN 模型模块

---

## 核心类

### `LightGCN`

```python
class LightGCN(nn.Module):
```

作用：定义 LightGCN 模型。

---

## 7.1 初始化函数

### `__init__(num_nodes, embedding_dim, num_layers, norm_adj)`

作用：

- 创建节点 embedding；
    
- 保存传播层数；
    
- 保存归一化邻接矩阵。
    

核心参数：

|参数|含义|
|---|---|
|`num_nodes`|节点数量|
|`embedding_dim`|节点向量维度|
|`num_layers`|LightGCN 传播层数|
|`norm_adj`|归一化邻接矩阵|

模型参数主要是：

```python
self.embedding = nn.Embedding(num_nodes, embedding_dim)
```

参数量大约是：

$$  
|V| \times d  
$$

---

## 7.2 获取节点表示

### `get_embeddings()`

作用：执行 LightGCN 的多层传播。

流程：

```text
e^(0) = 初始节点 embedding
e^(1) = A_norm e^(0)
e^(2) = A_norm e^(1)
...
e^(K) = A_norm e^(K-1)
```

最终取平均：

$$  
z_u=  
\frac{1}{K+1}  
\sum_{k=0}^{K}e_u^{(k)}  
$$

返回：

```python
final_emb
```

也就是所有节点最终的表示矩阵：

```text
[num_nodes, embedding_dim]
```

---

## 7.3 边打分函数

### `score_edges(edge_pairs, embeddings)`

作用：计算候选边 ((u,v)) 的得分。

公式：

$$  
score(u,v)=z_u^Tz_v  
$$

分数越高，越可能有边。

输入：

```python
edge_pairs
embeddings
```

输出：

```python
scores
```

# 8. `metrics.py`：评价指标模块

---

## 8.1 AUC / AP 评价

### `evaluate_auc_ap(model, pos_edges, neg_edges, device)`

作用：在固定正负样本上计算：

```text
AUC
AP
```

流程：

```text
1. 拼接正边和负边
2. 正边标签为 1，负边标签为 0
3. 用模型打分
4. sigmoid 转成概率
5. 计算 AUC 和 AP
```

AUC 衡量：

> 模型是否能把真实边排在负边前面。

AP 衡量：

> 正样本是否整体排得靠前。

---

## 8.2 Top-K 排名评价

### `evaluate_ranking(model, train_edges, test_edges, num_nodes, device, k)`

作用：计算：

```text
Precision@K
Recall@K
NDCG@K
```

对每个用户 (u)，候选集合是：

$$  
C(u)=V-{u}-N_{train}(u)  
$$

也就是排除：

```text
1. 用户自己
2. 训练集中已经是好友的人
```

然后对候选人排序，取 Top-K。

指标含义：

|指标|含义|
|---|---|
|`Precision@K`|推荐 K 个人中有多少是真实测试好友|
|`Recall@K`|测试好友中有多少被 Top-K 找到|
|`NDCG@K`|考虑命中好友的排名位置|

# 9. `trainer.py`：训练流程模块

这是整个工程的核心控制模块。

---

## 9.1 构造训练样本

### `_make_train_epoch_data(train_edges, train_neg_edges)`

作用：

```text
1. 拼接训练正边和训练负边
2. 正边标签为 1
3. 负边标签为 0
4. 打乱顺序
```

返回：

```python
train_pairs, train_labels
```

---

## 9.2 单轮训练

### `_train_one_epoch(...)`

作用：训练一个 epoch。

流程：

```text
1. 按 batch_size 切分训练样本
2. 获取 LightGCN 节点表示
3. 对 batch 中的边打分
4. 计算 BCEWithLogitsLoss
5. 反向传播
6. 更新参数
```

损失函数：

$$  
L=BCEWithLogitsLoss(score(u,v), y_{uv})  
$$

其中：

```text
y = 1 表示真实边
y = 0 表示负样本边
```

---

## 9.3 完整训练流程

### `run_train(cfg)`

作用：完成从数据到结果的完整实验。

这是最重要的主函数。

流程：

```text
1. 固定随机种子
2. 创建输出目录
3. 选择 CPU / GPU
4. 读取 Facebook 边数据
5. 构造 train / val / test 划分
6. 保存 split 文件
7. 构建 LightGCN 归一化邻接矩阵
8. 初始化 LightGCN 模型
9. 设置 optimizer 和 loss
10. 开始训练循环
11. 每个 epoch 动态采样训练负边
12. 在验证集上计算 AUC/AP
13. 根据 Val AP 保存最好模型
14. early stopping
15. 加载最好模型
16. 在测试集上计算最终指标
17. 保存实验结果
```

---

## 9.4 Early Stopping 逻辑

使用验证集 AP 作为选择标准：

```python
if val_ap > best_val_ap:
    保存模型
else:
    patience_counter += 1
```

如果连续 `patience` 轮没有提升：

```text
停止训练
```

目的：

> 防止过拟合，并节省训练时间。

# 13. 这个架构的优点

## 1. 更容易维护

原来所有代码都在一个文件里，数据、模型、训练、评价混在一起。

现在拆分后：

```text
想改模型 → 改 model.py
想改评价指标 → 改 metrics.py
想改数据划分 → 改 data_utils.py
想改训练流程 → 改 trainer.py
```

---

## 2. 更适合小组协作

做 MLP 的组员可以复用：

```text
data_utils.py
graph_utils.py
metrics.py
```

只需要新增：

```text
feature_utils.py
mlp_model.py
train_mlp.py
```

这样两个人可以共用同一套：

```text
数据划分
负样本
评价指标
```

保证公平性。

---

## 3. 更容易扩展

之后可以继续加入：

```text
GCN
GraphSAGE
GAT
BPR Loss
Hard Negative Sampling
MLP baseline
```

只需要新增模块，不需要重写整个工程。

---

# 14. 一句话总结

> 这个工程的核心架构是：`train.py` 作为入口，`trainer.py` 控制完整实验流程，`data_utils.py` 负责数据划分和负采样，`model.py` 定义 LightGCN，`metrics.py` 负责评价指标，其他模块提供配置和工具函数。整体设计目标是让数据、模型、训练和评价分离，方便复现实验、协作开发和后续扩展。
